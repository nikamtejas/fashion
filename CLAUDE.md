# LuxeLoom — Master Context

You are building LuxeLoom, a production-grade fashion e-commerce platform for the Indian market with a modern, unique, premium UI.

**Project layout (deviates from the original single-app prompt — see "Architecture deviations" below): two independently runnable/deployable projects at the repo root:**
- `backend/` — Express + TypeScript + Mongoose REST API, its own `package.json`, own port (default `4000`).
- `frontend/` — Next.js 14+ App Router app, its own `package.json`, own port (default `3000`), talks to the backend only over HTTP (`NEXT_PUBLIC_API_URL`).
- A thin root `package.json` (npm workspaces) exists only for convenience (`npm run dev` at root runs both); each folder also runs standalone (`cd backend && npm run dev`, `cd frontend && npm run dev`).

Frontend: TypeScript, Tailwind CSS, Framer Motion for animations, heavily customized shadcn/ui-style base components, Zustand for client state. Backend: Express + Mongoose as the ODM. The database is MongoDB Atlas (live cloud cluster) — connect via `MONGODB_URI` in the backend's environment, define proper Mongoose schemas with indexes on frequently queried fields (product slug, category, order status, user email, pincode), and never store secrets in code. All product and generated images are stored in Cloudinary — use the official Cloudinary Node SDK from the backend, upload via signed server-side uploads, store the `public_id` and `secure_url` in MongoDB, organize assets into folders per product, and always serve images through Cloudinary transformation URLs (f_auto, q_auto, responsive widths) for performance. Authentication is a custom JWT issued by the backend in an httpOnly cookie, with email OTP and Google OAuth (manual flow, not a Next.js-coupled library), and role-based access for CUSTOMER and ADMIN — see "Architecture deviations". PDF generation uses pdfkit or @react-pdf/renderer, run on the backend. Maps use Leaflet with OpenStreetMap on the frontend.

## Architecture deviations from the original prompt (decided during the build, not defaults)

- **Two separate apps instead of one Next.js app with API routes.** The user explicitly asked for a `backend/` and `frontend/` split that can be deployed and run independently (2026-07-12). A single Next.js app cannot satisfy that — `app/api` routes and pages always ship together as one process/deploy. So the backend is a standalone Express API.
- **Custom JWT auth instead of NextAuth.js.** NextAuth is Next.js-native; its encrypted session cookie is awkward to issue/verify from a separate Express process. The backend issues its own JWT in an httpOnly cookie (email OTP login, plus a manually-implemented Google OAuth2 authorization-code flow); the frontend calls backend auth endpoints and reads the current user from `GET /api/auth/me`. This exact tradeoff was already hit and solved the same way earlier in this project's history.

The design language is an editorial fashion-magazine aesthetic: a large serif display font such as Fraunces or Playfair Display paired with Inter, an ivory background (#FAF7F2), ink black (#141414), a burnt sienna accent (#C15B3C), muted sage for success states, generous whitespace, oversized product imagery, asymmetric editorial grids rather than uniform cards, micro-interactions everywhere (hover reveals, image zoom, magnetic buttons, smooth page transitions, skeleton loaders), a warm charcoal dark mode, and a fully responsive mobile-first PWA-installable build.

External integrations are Google Gemini 2.x image API (`GEMINI_API_KEY`) for AI photo generation, Razorpay (`RAZORPAY_KEY_ID`, `RAZORPAY_KEY_SECRET`) for online payments, Snapmint (`SNAPMINT_MERCHANT_ID`, `SNAPMINT_API_KEY`) for EMI, Blue Dart / DHL India (`BLUEDART_LICENSE_KEY`, `BLUEDART_LOGIN_ID`) for shipping, and Cloudinary (`CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, `CLOUDINARY_API_SECRET`). Every external call goes through a service layer in `/lib/integrations/` with timeouts, retries, and a global `INTEGRATIONS_MOCK=true` flag that returns realistic mock responses so the entire app works end-to-end in development without real keys. Provide a complete `.env.example`.

Ground rules: every milestone must end with the app running via `npm run dev` with zero TypeScript errors; seed the Atlas database with 20 realistic fashion products across Men, Women, Accessories and Footwear plus 3 store locations with real Indian lat/lng and pincodes; commit after each milestone with a descriptive message; always show me a short plan before implementing.
